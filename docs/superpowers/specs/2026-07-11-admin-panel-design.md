# /admin content-editing panel — design

Date: 2026-07-11
Status: approved by user, ready for implementation plan

## Purpose

Annabelle (non-technical site owner) currently edits her site's text by
opening `content.js` in Notepad and hand-editing a JavaScript object —
workable, but risky (a missing quote/comma breaks the page) and requires
her to know where each piece of text lives in the file. This project adds
a password-protected `/admin` panel where she edits the same content
through a plain web form, and hitting "Opslaan" publishes the change
immediately — no FTP, no code.

Confirmed with the user:
- Yourhosting (the target shared host) supports PHP, so `/admin` can write
  directly to the live site instead of only generating a file to
  re-upload.
- Scope for v1 is **text only**. Photos, colors, and layout stay on the
  existing FTP + Notepad workflow documented in `HANDLEIDING.md`.

## Chosen approach

A single custom PHP admin panel (no framework, no database, no build
step), consistent with the rest of the site's hand-crafted, dependency-free
approach. Rejected alternatives: splitting the form across multiple PHP
pages (unnecessary — the whole form fits on one scrollable page), and
dropping in an existing flat-file CMS (adds a new system to learn and
maintain for both Annabelle and Micha, and doesn't map onto the existing
`content.js` structure).

## Architecture

### Splitting `content.js`

Today `content.js` mixes the editable data (`SITE_CONTENT`) with the
rendering logic (`getContentValue`, `applyContent`, the `DOMContentLoaded`
listener). Once `/admin` can rewrite this file automatically, the logic
must live somewhere the admin panel never touches, or a future save could
silently wipe it out.

- `content.js` becomes a **pure, generated data file**: only
  `const SITE_CONTENT = {...};`, preceded by a comment noting it is
  auto-generated by `/admin` and should not be hand-edited once `/admin`
  is in use.
- `render.js` (new file) gets `getContentValue`, `applyContent`, and the
  `DOMContentLoaded` listener that calls it. Behavior is identical to
  today — this is a pure refactor, not a visible change.
- `index.html` script order becomes: `content.js` → `render.js` → `nav.js`
  (nav.js also reads `SITE_CONTENT` directly, so it must load after
  content.js, same as today).

### Source of truth

`admin/data/content.json` is the canonical data store — a JSON document
with the same shape as today's `SITE_CONTENT` object. It's seeded once
during implementation with the exact values currently in `content.js`
(including the `[INVULLEN]` placeholders), so the admin form is pre-filled
with everything that's already live.

On every save:
1. PHP validates and sanitizes the submitted form fields (see Validation).
2. The previous `content.json` is copied to
   `admin/data/backups/content-YYYYMMDD-HHMMSS.json`. Backups older than
   the 10 most recent are pruned.
3. The new data is written to `admin/data/content.json`.
4. `content.js` is regenerated from it:
   `const SITE_CONTENT = <json_encode(data, pretty-print)>;` — using
   `json_encode` means any character she types (quotes, `<`, backslashes,
   emoji) is safely escaped; there is no way a form submission can break
   the JS file or inject a script into it.

### File layout (new/changed files)

```
uitvaartendan/
├── content.js                  (changed: now generated, data-only)
├── render.js                   (new: logic moved out of content.js)
├── admin/
│   ├── index.php               (new: login form + dashboard, session-gated)
│   ├── save.php                (new: validates + writes content.json/content.js)
│   ├── wachtwoord.php          (new: change-password screen)
│   ├── logout.php              (new: destroys session)
│   └── data/
│       ├── config.php          (new: bcrypt password hash constant)
│       ├── content.json        (new: seeded from current content.js)
│       ├── .htaccess           (new: `Require all denied` — blocks all
│       │                        direct HTTP access to this folder)
│       └── backups/            (new: rolling save history, empty at deploy)
```

## Authentication & security

- Single shared password (Annabelle only needs one). Stored as a
  bcrypt hash (`password_hash`/`password_verify`) in `admin/data/config.php`
  — never stored or logged in plaintext.
- PHP session-based login (`session_start()` with `httponly`, `samesite=Lax`,
  and `secure` when served over HTTPS). No "remember me" — logs out when
  she closes the browser, which is fine for how infrequently this will be
  used.
- Failed-login throttling: a small file (`admin/data/lockout.json`, itself
  covered by the same `.htaccess` deny rule) tracks failed attempts. After
  5 consecutive failures, login is locked for 5 minutes. Resets on success.
- CSRF token embedded in the form and checked in `save.php` and
  `wachtwoord.php`.
- `admin/data/.htaccess` denies all direct web requests into that folder
  (`Require all denied`, with a legacy `Deny from all` fallback for older
  Apache) — defense in depth so `content.json`, `config.php`, or a stray
  `.bak` file can never be fetched directly, even if PHP execution were
  ever misconfigured.
- All values are HTML-escaped when redisplayed into the admin form
  (`htmlspecialchars`) and JSON-escaped when written into `content.js`
  (`json_encode`) — two different contexts, two different escaping
  strategies, both handled.
- Initial password is set during implementation and given to Micha
  out-of-band (in chat) once; Annabelle is expected to change it via the
  "wachtwoord wijzigen" screen the first time she logs in.

## Admin UI

One page (`/admin/`), grouped into the same sections Annabelle already
recognizes from `content.js` and the existing HANDLEIDING.md: Bedrijfsgegevens,
Menu, Home, Over mij, Diensten (including the repeatable pricing-package
item lists, with add/remove rows via a small vanilla-JS helper — no
framework), Contact, Footer. Fields needing attention (`[INVULLEN]`) are
visually flagged. One "Alles opslaan" button submits the whole form as a
single POST; a full-page reload shows a clear "Opgeslagen!" confirmation
banner. No AJAX/JS-fetch requirement — works even if something in her
browser blocks scripts.

## Error handling

- Server-side validation rejects a save that's missing a required field
  (e.g. company phone number) and re-renders the form with the offending
  field highlighted and everything else she typed preserved — she never
  loses work to a validation error.
- If `content.json` or `content.js` can't be written (e.g. a permissions
  problem on the host), `save.php` shows a plain-Dutch error message and
  does **not** touch the previous backup, so the live site keeps working
  off the last good save.
- Wrong password shows a generic "Onjuiste inloggegevens" message (no
  hint whether the password was close) plus the lockout behavior above.

## Documentation updates

`HANDLEIDING.md` is updated so `/admin` is presented as the primary way to
edit text, with a new short section explaining the login URL, the
password-change screen, and what to do if she forgets her password
(ask Micha, who can reset the hash in `admin/data/config.php` via FTP).
The existing Notepad/`content.js` instructions are kept only as a
documented fallback, with an explicit warning that hand-editing
`content.js` directly will be overwritten the next time she saves via
`/admin`.

## Testing / verification plan

Since there's no PHP available in the sandboxed dev environment used so
far in this project (previous verification relied on `npx serve`, a
static file server), verification needs a real PHP server:
- `php -S localhost:8000` from the project root (PHP's built-in dev
  server) to exercise login, save, backup rotation, and lockout end to
  end.
- Automated checks: login with wrong password 5x → confirms lockout;
  correct password → dashboard loads pre-filled with seeded content;
  edit a field, save → confirms `content.json` and `content.js` updated,
  backup file created, public `index.html` reflects the new text on
  reload; submit a value containing `"`, `<`, and a backslash → confirms
  it round-trips correctly with no broken page or console error; direct
  HTTP request to `admin/data/content.json` and `admin/data/config.php`
  → confirms both return 403 (blocked by `.htaccess`).
- Manual pass in a browser (via Playwright, same approach used earlier in
  this project) covering the full login → edit → save → logout flow, and
  a check that the public site still renders correctly after a save.

## Out of scope (v1)

- Photo uploads, color/theme editing, and any layout changes — stay on
  the existing FTP workflow.
- Multi-user accounts / roles — single shared password only.
- Password reset via email — no email-sending infrastructure on this
  static-turned-mostly-static host; reset is a manual step via FTP.
